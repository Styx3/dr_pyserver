# dr_pyserver — Dungeon Runners server (Python port)

A Python 3.12 / `asyncio` reimplementation of a game server for **Dungeon Runners**,
a discontinued free-to-play MMO built on the Lineage 2 network engine. The goal is
to bring the game back online by speaking the original client's wire protocol
exactly — byte for byte.

This is a **reverse-engineering / preservation project**: a Python port of an
earlier C# server, cross-checked against the original client binary and its
extracted content. Most of the hard-won protocol knowledge lives in
**[`bible.md`](bible.md)** — see [Documentation](#documentation).

> **Status — work in progress.** Auth, character creation, world/zone spawn,
> movement, inventory/equipment, merchants, quests, chat, groups, and
> client-authoritative combat relay work against the real client; many systems are
> still partial. This repository is **source only** — no game client, art, or
> content database is included (see [Database](#database)).

---

## Requirements

- **Python 3.12**
- The original **Dungeon Runners client** (`Client666`) to actually connect. You
  must supply this yourself; it is not distributed here.
- A **content database** — also not distributed (see [Database](#database)).
- No C toolchain is needed for the server itself. The *optional* client-side hook
  under `client_hook/` needs a 32-bit MinGW cross-compiler — see
  [`client_hook/README.md`](client_hook/README.md).

## Setup

```bash
# 1. Clone
git clone git@github.com:Styx3/dr_pyserver.git
cd dr_pyserver

# 2. Create and activate a virtual environment
python -m venv .venv
source .venv/bin/activate             # Windows: .venv\Scripts\activate

# 3. Install dependencies
pip install -r requirements.txt       # pycryptodome, PyYAML
pip install pytest pytest-cov         # optional: for the test suite

# 4. Create your local config
cp config.example.yaml config.yaml    # config.yaml is git-ignored (local only)
```

Edit `config.yaml` to set the addresses/ports the server binds to. It holds
**static** configuration: bind IPs, ports, the client-extracted encryption keys,
and spawn defaults. **Dynamic** gameplay tunables live separately in the
`server_settings` DB table (overlaid at boot from an optional `server.cfg`).

## Database

The server reads a SQLite database at the path in `config.yaml` (`database_path`,
default `Database/dungeon_runners.db`). **It is not shipped with this repo** — it is
large and contains extracted game content. There are two layers:

- **Player/account tables** (`accounts`, `characters`, `character_inventory`,
  `character_equipment`, `character_skills`, `character_quests`,
  `character_checkpoints`, `server_settings`, …) are **created automatically on
  first boot**. `drserver/db/game_database.py` runs `CREATE TABLE IF NOT EXISTS`
  plus lightweight column migrations, so pointing `database_path` at a new/empty
  file is enough to register an account and create characters.
- **Static content tables** (50+ tables: zones, creatures, NPCs, items, quests,
  skills, merchants, stat pools, …) are **not** generated by the schema. They are
  imported from the original client's extracted content and a reference server
  dump. Without them the client fails to load a world ("error talking to the
  server").

To populate content, use the importers in `scripts/` — e.g.
`scripts/restore_database.py` (extracts the content DB from a `DR_Server.zip`
reference dump and merges in live player rows) and
`scripts/rebuild_content_tables.py` (rebuilds tables from extracted `.gc` files).
Both need you to supply the external source data; see each script's header for the
expected paths.

## Running the server

```bash
python -m drserver
```

Boot order (`drserver/__main__.py`): load `config.yaml` → open the SQLite DB (creating
player tables if absent) → apply the `server_settings` overlay → start the auth and
game servers concurrently. If `admin_panel_enabled` is set, a browser admin
dashboard is served too.

| Listener | Default port | Status |
|----------|:---:|--------|
| Auth server | 2110 | Working |
| Game server | 2603 | Working |
| Queue server | 2606 | Stubbed — handoff via an IP→login map (`core/sessions.queue_bridge`) |
| Admin panel | 8080 | Optional (`admin_panel_enabled`; login needs an `is_admin=1` account) |

Point the client at the auth server's address/port to connect.

## Testing

```bash
pytest                                             # full suite
pytest -v                                          # verbose, per test
pytest tests/test_auth_crypto.py                   # a single file
pytest --cov=drserver --cov-report=term-missing    # coverage
```

Tests run from the repo root and resolve paths relative to the repo — no absolute
paths. Integration tests that need the extracted client content skip themselves
automatically when it is absent. Every ported protocol handler ships with a
round-trip byte test; match the AAA style and naming in `tests/`.

## Documentation

- **[`bible.md`](bible.md) — the Sync Bible.** The single most important document
  here: ~2,700 lines distilling the reverse-engineering of the client's networking
  and synchronization model, gathered from Ghidra/x64dbg (static **and** live) plus
  packet captures. **Read it before porting or changing any protocol behavior.** It
  is organized by **evidence tier** — `[T0]` = proven against the client binary /
  live trace, down to `[T3]` = fabricated or unproven lore to distrust — so you can
  always tell fact from guess. Highlights:
  - Time & the message-rate contract; RNG determinism & the seed model
  - The **zero-tolerance HP-synch compare** — why the server must never send an HP
    value that contradicts the owning client
  - The authority model: *reproduce shared state, relay simulated state* — combat is
    **client-authoritative**; the server mirrors and relays, it does not re-simulate
  - Per-player / per-group zone instancing; loot ownership
  - A build checklist for a desync-free server, an evidence catalogue of client
    functions/fields, the quest protocol, and combat-balance data
- **`docs/`** — supporting reference dumps: client symbol tables
  (`docs/client/*.txt`) and maze-parity captures (`docs/maze_parity/*.txt`).
- **[`client_hook/README.md`](client_hook/README.md)** — the optional client-side
  `d3d9.dll` proxy/telemetry hook used during combat RE.

The "golden gotchas" every contributor trips on at least once — little-endian
everywhere, the Blowfish little-endian word-order trap, the 256× fixed-point
("drfloat") scaling of positions and HP/MP, owner-only components that must never
be sent to other clients, and verbatim client session IDs — are all documented in
`bible.md`.

## Project layout

| Package | Responsibility |
|---|---|
| `drserver/net/` | TCP servers, framing, auth handshake, channel dispatch, spawn, movement, inventory, equipment, chat |
| `drserver/managers/` | Combat, monsters, NPCs, merchants, quests, groups, social, loot, duels, world entities, zones, bling gnome |
| `drserver/data/` | GCObject serialization + factory, character/class models, item-stat data |
| `drserver/db/` | SQLite repositories (account, character) + connection and schema |
| `drserver/core/` | config, settings, sessions, logging, dispatcher, clock |
| `drserver/util/` | byte IO (LE reader/writer), crypto (Blowfish/DES), zlib, maze |
| `drserver/combat/` | RNG (Mersenne Twister / xorshift) |
| `drserver/admin/` | Browser-based admin dashboard |
| `scripts/` | DB restore/rebuild + protocol-debugging utilities |
| `client_hook/` | Optional client-side hook (C) for combat telemetry during RE |

## Contributing

- **Verify against the client, don't assume.** Wrong byte layouts fail silently —
  the client just drops the packet. Mark uncertain details `# UNVERIFIED` and
  confirm against the client binary / PDB.
- **Port with a test.** Every handler gets a round-trip pytest.
- PEP 8, type annotations on signatures, `black` / `ruff`; keep modules small and
  per-feature rather than growing monoliths.
